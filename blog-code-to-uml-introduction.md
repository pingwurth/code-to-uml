# Vibe Coding 时代，不想看代码又必须理解代码怎么办？

**先看成果：**

我用 AI 分析了一个 2069 行的 Python 文件，10 分钟生成了这份报告：

![Code-To-UML 演示](docs/images/demo_zh.gif)

报告包含 10 个维度：
- 概览：文件定位、复杂度评分（12/12）
- 结构：导入、常量、类/函数定义
- 对象：主要类、数据结构、关键函数
- 架构：分层图、模块边界、依赖方向
- 流程：代理主循环、工具执行、错误处理
- 调用：函数调用链、跨边界调用关系
- 数据流：数据流动、状态转换、持久化机制
- 代码：核心函数片段、设计意图
- 原理：系统设计、机制、权衡
- 指南：如何阅读、运行、调试、扩展

**这就是 Code-To-UML。**

---

## 一、痛点：Vibe Coding 的"副作用"

Cursor、Copilot、Claude Code……AI 帮你写了代码，但：

- 500 行 AI 代码，你还是要逐行读
- 新项目接手，你还是要花几天摸架构
- 线上 Bug，你还是要逐行打断点
- 代码审查，你还是要反复切文件

**AI 帮你写了代码，但没人帮你读代码。**

你不想看代码，但你必须理解代码。

---

## 二、解决方案：让 AI 帮你"读代码"

如果 AI 能写代码，为什么不能**读代码**？

**核心理念：你给代码，AI 分析，工具渲染。**

```
你: "分析这个项目" → AI 阅读代码 → 生成 .ctu 数据 → 渲染交互式报告
```

你不需要手动画图，AI 帮你完成所有分析。

---

## 三、实战拆解：claude-code-demo-analysis

让我用一个真实案例，展示 Code-To-UML 到底能做什么。

### 背景

我有一个 `claude_code_demo.py`，2069 行，实现了 AI 代理的完整架构：
- 任务系统、工作树隔离、技能加载
- 消息总线、钩子系统、上下文压缩
- 错误恢复、后台任务、Cron 调度、MCP 集成

**问题：代码太复杂，怎么快速理解？**

### 解决方案

在 AI 工具中输入：

```
/code-to-uml 请分析 demo/claude_code_demo.py 文件，并生成基于 UML 的、格式统一的 HTML 报告
```

### 生成结果

#### 1. 概览：一眼看清复杂度

报告开头直接给出复杂度评分：

| 维度 | 评分 | 说明 |
|------|------|------|
| 规模 | 2 | 2069 行代码，大型文件 |
| 符号数量 | 2 | 30+ 有意义的类/函数 |
| 调用关系 | 2 | 跨模块、多线程、异步回调 |
| 状态/数据 | 2 | 持久化状态、消息总线、协议状态 |
| 分支/错误路径 | 2 | 重试、恢复、回退、并发控制 |
| 子系统数量 | 2 | 10+ 独立子系统 |

**复杂度评分：12/12（高复杂度）**

不用读代码，就知道这个文件"不简单"。

#### 2. 架构：一张图看清分层

报告生成了架构分层图：

```
┌─────────────────────────────────────────────────────────────┐
│                    CLI 入口层                                │
│ __main__ block, 用户交互, cron_autorun_loop                 │
└─────────────────────────┬───────────────────────────────────┘
                          │ 调用
┌─────────────────────────▼───────────────────────────────────┐
│                    代理循环层                                │
│ agent_loop, prepare_context, call_llm, 错误恢复             │
└─────────────────────────┬───────────────────────────────────┘
                          │ 使用
┌─────────────────────────▼───────────────────────────────────┐
│                    工具执行层                                │
│ BUILTIN_HANDLERS, 工具调度, 权限检查, 后台任务               │
└─────────────────────────┬───────────────────────────────────┘
                          │ 依赖
┌─────────────────────────▼───────────────────────────────────┐
│                    子系统层                                  │
│ 任务系统, 工作树, 技能加载, MCP, 消息总线, Cron             │
└─────────────────────────┬───────────────────────────────────┘
                          │ 使用
┌─────────────────────────▼───────────────────────────────────┐
│                    基础设施层                                │
│ 文件系统, Git, Anthropic API, 环境变量                      │
└─────────────────────────────────────────────────────────────┘
```

**依赖方向：上层依赖下层，下层不依赖上层。**

不用读代码，就知道系统的分层结构。

#### 3. 流程：一张图看清主循环

报告生成了代理主循环的流程图：

```
开始
  ↓
注入 Cron 任务
  ↓
收集后台通知
  ↓
检查 todo 提醒
  ↓
准备上下文
  ↓
组装工具集
  ↓
调用 LLM
  ↓
prompt-too-long?
  ├─ 是 → 执行 reactive_compact → 返回循环
  └─ 否 → 继续
  ↓
max_tokens?
  ├─ 是 → 升级 max_tokens → 返回循环
  └─ 否 → 继续
  ↓
执行工具调用
  ↓
处理错误恢复
  ↓
循环...
```

**关键决策点一目了然。**

不用读代码，就知道系统的运行逻辑。

#### 4. 调用关系：一张图看清函数依赖

报告生成了函数调用链图，展示：
- `agent_loop` 调用 `prepare_context`、`call_llm`、`execute_tool`
- `prepare_context` 调用 `compress_context`、`inject_cron_tasks`
- `execute_tool` 调用 `BUILTIN_HANDLERS`、`check_permissions`

**跨边界调用关系清晰可见。**

不用读代码，就知道函数之间的依赖关系。

## 四、核心特性
